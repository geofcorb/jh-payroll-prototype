How to use this tool

This tool compares a single employee's payroll for one pay run between the legacy SAP system and the future Workday system, component by component, and explains the likely cause of each discrepancy.

Workflow

  • Find employee — search by name, ID, or department and select one.
  • Pick a pay period — the strip shows net variance per period; click one to drill in.
  • Read the summary cards (gross, deductions, net, employer cost), the reconciliation bars, then the component table and discrepancy breakdown.
  • Click a cause chip to filter the table to just that type of discrepancy.

How variance & net impact read

Variance = Workday − SAP. Green means Workday is higher, red means lower. Net impact shows how that line moves take-home pay (earnings add to net; deductions and taxes subtract; employer items don't affect net).

Discrepancy causes (auto-detected)

Rate / amount differenceSame component, materially different value — a rate/salary/formula updated in one system only. Tax configuration differenceA withholding differs beyond rounding — tax setup, elections, or taxable base differ. Wage-type mapping gapComponent exists in one system only — an unmapped wage type or unconfigured pay component. Proration differenceDifferent worked/paid days — mid-period hire, termination, or leave. Retroactive adjustmentA retro line in one system only — expected to differ during parallel runs. Rounding differenceAgreement within a few cents — different rounding rules.

Loading your own data

Click Export / template to download the current data as CSV and see the exact format, then Import CSV to load real SAP and Workday extracts. One row per component per system. Columns:

system, employee_id, employee_name, department, location, pay_group, period_id, period_label, period_start, period_end, pay_date, period_days, component_code, component_name, category, amount, retro, worked_days

  • system = SAP or Workday.
  • category = earning, pre_tax, tax, post_tax, or employer.
  • Use the same component_code in both systems so the tool can pair them; an unpaired code is reported as a mapping gap.
  • retro = Y for retroactive lines; worked_days drives proration detection (leave blank for full-period).

All processing happens locally in your browser — no data leaves the page.
